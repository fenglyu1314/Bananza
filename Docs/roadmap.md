# Bananza 交付路线图 (Roadmap)

> **本文件性质**：非规范性（non-normative）的阶段排期参考，不属于 OpenSpec spec。
>
> - 本文件仅描述"大致按什么顺序把玩法做出来"，用于指导 change 的排期与优先级。
> - **本文件中的任何描述都不构成规范约束**。真正的 requirement / scenario / 验收条件以 `openspec/specs/` 下的 spec 为准，具体工作拆分与变更决策以 `openspec/changes/` 下的 change 为准。
> - 本文件本身的调整**不走 change 流程**，直接编辑即可；调整时请在文末"变更记录"追加一行。
> - 当实际开发中发现 roadmap 与真实节奏偏离较大时，MUST 及时修订本文件，避免其变为过期信息。

## 指导原则

- 每个阶段只描述**该阶段结束时玩家应该能体验到什么**（粗验收），不描述实现细节。
- 阶段之间**尽量保持可玩**：每完成一个阶段，都应该有一个"虽然内容少但能跑起来"的可测试版本。
- 阶段内部的具体能力、任务拆分、验收标准由该阶段对应的若干 change 各自定义。
- 阶段顺序不是绝对锁死的，但调整顺序需要先更新本文件。

## 阶段划分

### P1 · 角色基础控制

**目标**：玩家可以在一个测试场景中自由操控主角，并对场景中的物体进行最基本的物理互动。

**包含的大致能力方向**：

- 角色移动、转向、跳跃等基础位移
- 相机跟随
- 单件物体的拾取与放下
- 物体投掷

**粗验收**：在一个用于测试的简单场景中，玩家可以走动、跳跃，走到一个物体旁边把它捡起来，走到别处把它放下或投掷出去，整个过程无需任何 UI、任务、存档。

---

### P2 · 世界与物体持久化

**目标**：玩家对世界所做的改变（搬动物体、破坏、状态切换等）在离开区域或重开游戏后 SHOULD 被正确恢复。

**包含的大致能力方向**：

- 可互动物体的运行时状态建模
- 场景切换 / 区域切换时的状态保持
- 本地存档与读档（仍然严格遵循 vision spec 中"单机离线"的约束）

**粗验收**：玩家把物体 A 从房间 X 搬到房间 Y，走远再回来，或者退出游戏再进入，物体 A 仍然在房间 Y。

---

### P3 · 任务系统

**目标**：玩家可以接收、进行、完成结构化的"捣蛋任务"，任务应作为引导探索的软性牵引，而非强制流程。

**包含的大致能力方向**：

- 任务定义与触发
- 任务进度追踪
- 任务奖励 / 完成反馈（遵循 vision spec 中"无失败、无惩罚"约束）
- 最小可用的任务相关 UI

**粗验收**：玩家在场景中能遇到至少一个可接取的任务，按提示完成目标行为后得到完成反馈；放弃或忽略任务不会导致任何惩罚。

---

### P4 · 双人控制

**目标**：在同一台设备上支持双人同屏协作进行 P1~P3 已有的所有玩法。

**包含的大致能力方向**：

- 本地双人输入（手柄 / 键鼠分配）
- 双角色相机策略
- 已有玩法在双人模式下的适配（拾取、投掷、任务、持久化等）

**粗验收**：两位玩家可以在同一台设备上同时操控各自的角色，在同一场景中进行 P1~P3 的全部行为，不会因双人模式出现明显冲突或退化。

---

### P5 · NPC

**目标**：场景中加入具有自主行为的 NPC，使世界更有生气，并成为"捣蛋"这一核心动词的主要反馈来源。

**包含的大致能力方向**：

- NPC 行为框架（巡逻、反应、被打扰后的行为等）
- NPC 与玩家、物体、任务系统的互动
- NPC 相关的情绪 / 表现反馈

**粗验收**：场景中存在可与玩家产生互动的 NPC，玩家的捣蛋行为能引起 NPC 的即时反应；NPC 的反应仅限 vision spec 中允许的"局部、可恢复"形式，不会导致玩家失败或进度损失。
