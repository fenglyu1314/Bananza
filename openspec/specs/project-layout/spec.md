# project-layout Specification

## Purpose

This capability defines the canonical directory layout and assembly definition (`.asmdef`) strategy for the Bananza Unity project. It prescribes where source code, ScriptableObject type definitions, asset instances, scenes, settings, and third-party plugins live under `Assets/`, and under what conditions new `.asmdef` files may be split out from the two baseline assemblies (`Bananza.Runtime`, `Bananza.Editor`). Downstream documents such as [AGENTS.md](../../AGENTS.md) and `README.md` defer to this spec rather than duplicating its content.

## Requirements
### Requirement: Top-level Asset Directory Layout

The project SHALL organize `Assets/` into a fixed set of top-level directories whose purpose is single and non-overlapping. Each directory MUST exist only when it contains files; empty directories MUST NOT be created solely to satisfy this requirement.

The canonical top-level directories under `Assets/` are:

- `Art/` — visual resources (textures, models, materials, animations, VFX)
- `Audio/` — audio resources
- `Prefabs/` — prefab instances
- `Scenes/` — scene files
- `Scripts/` — C# source code (see *Scripts Subtree Layout*)
- `Settings/` — URP render pipeline assets and input settings
- `Data/` — ScriptableObject asset instances (`.asset`), grouped by business module (NOT C# type definitions)
- `ThirdParty/` — external plugins, each isolated in its own subfolder with its own `.asmdef`
- `Tests/` — reserved for test assemblies (optional, created when tests are first introduced)

#### Scenario: Adding a new SO asset instance

- **WHEN** a designer creates a new `PropData` ScriptableObject instance
- **THEN** the `.asset` file MUST be placed under `Assets/Data/Props/` and the corresponding `.meta` MUST be generated by the Unity Editor

#### Scenario: Adding a third-party plugin

- **WHEN** a developer imports a new third-party plugin
- **THEN** the plugin MUST be placed under `Assets/ThirdParty/<PluginName>/` and MUST carry its own `.asmdef` to isolate it from project code

#### Scenario: Attempting to mix code and data in `Assets/Data/`

- **WHEN** a developer is tempted to place a `.cs` ScriptableObject definition under `Assets/Data/`
- **THEN** the placement MUST be rejected — `Assets/Data/` only holds `.asset` instances; type definitions live under `Assets/Scripts/Runtime/<Module>/` per the *SO Type Definitions Co-located with Business Modules* requirement

### Requirement: Scripts Subtree Layout

All C# source code SHALL live under `Assets/Scripts/`, partitioned into two roots that correspond one-to-one with the two project `.asmdef`s:

- `Assets/Scripts/Runtime/` — managed by `Bananza.Runtime.asmdef`, `rootNamespace = Bananza`
- `Assets/Scripts/Editor/` — managed by `Bananza.Editor.asmdef`, `rootNamespace = Bananza.Editor`

The legacy location `Assets/Editor/` MUST NOT be re-created; all editor-only scripts MUST live under `Assets/Scripts/Editor/`.

Within `Runtime/`, the recognized first-level modules are `Core`, `Gameplay`, `NPC`, `AI`, `UI`. Within `Editor/`, the recognized first-level modules are `Inspectors`, `TaskGraph`, `Tools`. Introducing additional first-level modules SHALL require updating this spec via a change proposal.

#### Scenario: Creating a new gameplay script

- **WHEN** a developer adds a new `PlayerController.cs` for the player module
- **THEN** the file MUST be placed at `Assets/Scripts/Runtime/Gameplay/Player/PlayerController.cs` and declare `namespace Bananza.Gameplay.Player`

#### Scenario: Creating a new editor inspector

- **WHEN** a developer adds a custom inspector for `PropData`
- **THEN** the file MUST be placed under `Assets/Scripts/Editor/Inspectors/` and declare a namespace rooted at `Bananza.Editor.Inspectors`

#### Scenario: Accidentally introducing `Assets/Editor/`

- **WHEN** any tool or developer attempts to create `Assets/Editor/` again
- **THEN** the action MUST be rejected; editor scripts MUST stay under `Assets/Scripts/Editor/`

---

### Requirement: Directory-Namespace One-to-One Mapping

Every sub-directory under `Assets/Scripts/Runtime/` and `Assets/Scripts/Editor/` SHALL correspond exactly to a C# namespace segment, so that splitting an assembly in the future requires only moving a folder — not renaming types.

The nesting depth under `Runtime/` MUST be at most two levels (`Runtime/<Module>/<SubModule>/`). Exceeding two levels is a signal that the affected module should be split into its own `.asmdef`.

Module boundaries MUST be respected: code belonging to one recognized module MUST NOT be placed under another module's directory (e.g., `Bananza.AI` code MUST NOT live under `Gameplay/`).

#### Scenario: Namespace derived from path

- **WHEN** a file lives at `Assets/Scripts/Runtime/AI/Brain/DecisionTree.cs`
- **THEN** its namespace MUST be `Bananza.AI.Brain`

#### Scenario: Depth limit exceeded

- **WHEN** a developer proposes creating `Assets/Scripts/Runtime/Gameplay/Props/Interactables/PickUp/PickUp.cs` (three levels below `Runtime/`)
- **THEN** the proposal MUST instead evaluate splitting `Props` (or a sibling) into its own `.asmdef` via a change proposal, rather than deepening the tree

#### Scenario: Code placed in the wrong module

- **WHEN** an AI-system script is accidentally placed under `Assets/Scripts/Runtime/Gameplay/`
- **THEN** the script MUST be moved to `Assets/Scripts/Runtime/AI/<SubModule>/` and its namespace updated accordingly before merge

---

### Requirement: SO Type Definitions Co-located with Business Modules

ScriptableObject **type definitions** (`.cs`) SHALL live next to the business module that uses them, NOT in a dedicated `Scripts/Runtime/Data/` root. ScriptableObject **asset instances** (`.asset`) SHALL live under `Assets/Data/<Module>/`, organizationally separated from their type definitions.

A dedicated `Bananza.Data` assembly MUST NOT be planned; SO type definitions follow their owning module's assembly.

#### Scenario: Placing `PropData.cs`

- **WHEN** a developer defines the `PropData` ScriptableObject type
- **THEN** `PropData.cs` MUST be placed under `Assets/Scripts/Runtime/Gameplay/Props/` (next to the prop runtime logic)

#### Scenario: Placing the corresponding asset instance

- **WHEN** the first `Apple.asset` instance of `PropData` is authored
- **THEN** the asset MUST be saved under `Assets/Data/Props/Apple.asset`

#### Scenario: NPC profile placement

- **WHEN** defining `NPCProfile.cs` that is consumed by both the NPC runtime and the AI brain
- **THEN** the file MUST be placed under `Assets/Scripts/Runtime/NPC/` OR `Assets/Scripts/Runtime/AI/Configs/` based on primary consumer, and documented in the commit message

### Requirement: Assembly Definition Baseline

The project SHALL maintain exactly two assemblies at baseline:

| asmdef | Path | rootNamespace | References |
|---|---|---|---|
| `Bananza.Runtime` | `Assets/Scripts/Runtime/` | `Bananza` | (none) |
| `Bananza.Editor`  | `Assets/Scripts/Editor/`  | `Bananza.Editor` | `Bananza.Runtime` |

Additional `.asmdef` files MUST NOT be created outside the split-triggering conditions defined in *Assembly Split Triggers*.

Dependency direction is strict and one-way:

- Runtime modules MUST NOT reference editor code.
- `Editor` MAY reference any `Runtime` assembly but the reverse MUST NOT occur.

#### Scenario: Baseline preserved

- **WHEN** inspecting a fresh checkout of the project
- **THEN** exactly two `.asmdef` files MUST exist: `Bananza.Runtime.asmdef` and `Bananza.Editor.asmdef`

#### Scenario: Runtime references Editor (forbidden)

- **WHEN** a pull request attempts to add `Bananza.Editor` to the `references` list of `Bananza.Runtime.asmdef`
- **THEN** the change MUST be rejected as a violation of the one-way dependency rule

---

### Requirement: Assembly Split Triggers

A new `.asmdef` SHALL be introduced ONLY when at least one of the following triggers is met, and the split MUST be proposed through an OpenSpec change (not committed ad-hoc):

1. A sub-module accumulates 15 or more `.cs` files.
2. Cross-module circular dependencies appear and cannot be resolved without assembly boundaries.
3. A third-party plugin is introduced (plugins always get their own `.asmdef` under `Assets/ThirdParty/<Plugin>/`).
4. A module requires platform scoping (Editor-only, Standalone-only, or mobile-only) via `includePlatforms` / `excludePlatforms`.
5. Editor tooling scripts (especially GraphView-style editors) reach 10 or more files.
6. Unit or integration tests are introduced — dedicated `Bananza.Tests.Runtime` / `Bananza.Tests.Editor` assemblies MUST be created with the Test Assemblies flag set.

When multiple triggers apply to the same module, the split MAY be combined into a single change.

#### Scenario: File count threshold crossed

- **WHEN** the file count under `Assets/Scripts/Runtime/AI/` reaches 15
- **THEN** an OpenSpec change proposing `Bananza.AI` as a dedicated assembly MUST be drafted before additional AI files are merged

#### Scenario: Introducing a third-party plugin

- **WHEN** the team decides to import DOTween
- **THEN** the plugin MUST be placed under `Assets/ThirdParty/DOTween/` and MUST ship with its own `.asmdef` named `DOTween` (or the vendor-provided name)

#### Scenario: First unit test added

- **WHEN** the first unit test file is created
- **THEN** an OpenSpec change MUST introduce `Bananza.Tests.Runtime.asmdef` (and `Bananza.Tests.Editor.asmdef` if editor tests are needed) with the Test Assemblies flag enabled

---

### Requirement: Standard Procedure for Adding a New `.asmdef`

When a split trigger is met, the new `.asmdef` SHALL be added following this procedure:

1. Create the `.asmdef` inside the target directory via the Unity Editor (Create → Assembly Definition) so that Unity generates the `.meta` automatically. AI assistants MUST NOT hand-create `.asmdef` or its `.meta`.
2. Name the assembly `Bananza.<Module>` (runtime) or `Bananza.Editor.<Tool>` (editor tooling).
3. Set `rootNamespace` to match the directory's namespace (per *Directory-Namespace One-to-One Mapping*).
4. Declare only **direct** dependencies in `references`. Circular references MUST be rejected.
5. For editor-only assemblies, set `includePlatforms = ["Editor"]`.
6. Commit using Conventional Commits: `chore(asmdef): split <Module> into Bananza.<Module>`.
7. Update this spec via an OpenSpec change (e.g., `split-<module>-asmdef`) in the same pull request.

#### Scenario: Splitting out `Bananza.Core`

- **WHEN** the team decides to extract `Bananza.Core` from `Bananza.Runtime`
- **THEN** a change named `split-core-asmdef` MUST be proposed, the `.asmdef` MUST be created inside `Assets/Scripts/Runtime/Core/` via the Unity Editor, and this spec MUST be updated to reflect the new baseline

#### Scenario: AI tries to hand-write `.meta`

- **WHEN** an AI assistant is asked to create a new `.asmdef` and tempted to also write its `.meta`
- **THEN** the assistant MUST refuse `.meta` creation, create only the `.asmdef`, and instruct the user to refresh the Unity Editor so the `.meta` is generated

---

### Requirement: Spec as the Single Source of Truth for Project Layout

The directory-and-assembly conventions defined by this spec SHALL be the single authoritative reference for the project. Downstream documents (e.g., [AGENTS.md](../../AGENTS.md), `README.md`) MUST defer to this spec rather than duplicating its contents.

Temporary or long-form human-readable guides MUST NOT duplicate normative content from this spec; if a digest is desired, it MAY summarize but MUST link back to this spec.

#### Scenario: AGENTS.md references layout rules

- **WHEN** AGENTS.md needs to describe code placement rules
- **THEN** AGENTS.md MUST refer to "the OpenSpec `project-layout` spec" rather than embedding the full rules or pointing to a specific file path under `openspec/`

#### Scenario: Conflict between docs and spec

- **WHEN** a discrepancy is discovered between this spec and any Docs/ file or README
- **THEN** this spec MUST be treated as authoritative and the other document MUST be updated to align (or to defer to this spec)
